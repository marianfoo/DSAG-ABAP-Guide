# Beteiligung an Open-Source-Software

This section describes the use case of participating in open source projects as a company. He thus addresses the second [Ausbaustufe]({{ site.baseurl }}/open-source/#ausbaustufen).

1. TOC
{:toc}

## Opportunities to participate in open source development

When you participate in open source projects, you actively influence their development. This can be done by providing coding sections that extend the functionality of the application, but it can also be much more simply an issue with an error scenario or the addition of documentation or tests. The majority of the ABAP projects developed in open source are managed in their free time by their maintainers, who are generally happy about any kind of participation. The barrier to entry is therefore very low.

Your participation also brings you some advantages: By participating in such external projects, you inevitably come into contact with other technological challenges, libraries and frameworks, which allows you to expand your skills. If you contribute coding, it will be subject to a code review, which allows you to receive and incorporate feedback. You can also participate in the open source project by testing and reviewing pull requests and thus increasingly familiarizing yourself with the codebase.

## Possible organizational hurdles

Participation in this form is regularly possible in the free time that many open source maintainers use for this purpose. Things look different during working hours. If your company wants to invest in the maintenance and expansion of open source projects that it may also use itself, then appropriate guidelines should be in place that the respective employees can follow. In addition to working hours, licensing and copyright issues also need to be clarified.

## Types of participation

Participation can take place in many different ways.

- __Opening issues__  
  The easiest way to get involved is to report bugs or submit feature requests. The vast majority of open source projects offer a tracker in which issues can be recorded and discussed with the community. If necessary, you can also offer to solve the problem yourself and submit a pull request. Make sure that your issues are well formulated and represent the problem in a reproducible way. If necessary, the project provides a template for structuring the issue.
- __Addition of documentation and tests__  
  Most open source projects include documentation on how to use the software and contribute to its development. As a user who has used the documentation to set it up himself, it is easy to close gaps and improve the documentation. In addition, there are often automated software tests that can be supplemented with additional test cases. In both cases, the actual coding is not yet touched, but the project itself is increasingly improved.
- __Performing bug fixes__  
  To actually get into the codebase, you can edit simple issues. Some open source projects offer issues with the label `good-first-issue` for exactly this use case, which are particularly suitable for getting started. Please pay particular attention to the contribution guidelines of the respective project when making coding changes. A useful extension is often the addition of a unit test to prevent the problem from recurring in the future.
- __Addition of new features__  
  The most comprehensive option to participate is to contribute your own features. In this case, you add new functionality to the open source project. You can already do this internally, independently of the project - but you should also make your extensions available in the project so that others can also benefit from them. Always create an issue first to discuss your desired extension with the maintainers and the community and to coordinate a possible implementation approach. Here too, the contribution guidelines are very important. You can then propose the extensions and respond to code reviews via a pull request.
