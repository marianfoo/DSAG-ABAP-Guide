# General documentation tips

1. TOC
{:toc}

The following section provides you with easy-to-use and useful tips on how you can handle documentation tasks effectively. The tips are not limited to technical documentation, but can be applied universally to documentation tasks.

## Golden documentation rules

### Documentation must be easy to understand (#1)
+ Write in active form and use short sentences.
- Use easy-to-understand and simple language.
- Write superficially and do not overload the reader with too many details.
- Only go into details if they offer the target group added value.
- Visualize information through the use of tables and graphics.

### Documentation must be easy to use (#2)
+ Write in a target group-oriented manner and use domain-specific technical language.
- Use a solution-oriented writing style that supports the reader in completing their tasks.
- Use modern documentation tools such as hyperlinks for semantic context navigation or reusable text containers.
- Ensure easy findability and easy access to the documentation for the target group.
- When using information and images, pay attention to avoiding data protection or compliance violations.
- Use documentation templates and provide reference documentation.
- Delete empty chapters and template texts that come from a template document (These will be out of date after the next template update, so you no longer need them!)

### Documentation must be specific and understandable (#3)
+ Depending on the target group and use, separate the documentation into different documentation artifacts, such as administration manual, requirements documents, technical documentation, etc. 
- Use a version management system for the documentation artifacts to easily track documentation changes made via the version history.

### Documentation must provide a meaningful value contribution (#4)
+ The documentation benefit must be greater than the costs of creating the document and keeping it up to date.
- Involve the authors and the readers for whom the documentation is written in the definition process of the documentation guidelines.
